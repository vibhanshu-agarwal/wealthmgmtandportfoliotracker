# Requirements Document

## Introduction

This feature modernizes the deployment and verification pipeline for the Wealth Management & Portfolio Tracker platform. The goal is to achieve high confidence before AWS deployment by introducing multi-stage Docker builds with AOT compilation and AWS Lambda Web Adapter support, consumer-driven contract testing via Pact, hardened Playwright E2E tests that run against containerized services, and a refactored GitHub Actions CI/CD pipeline that builds and verifies container images end-to-end. The feature also optimizes local developer machine resources by shifting heavy builds into Docker and eliminating redundant build steps.

## Glossary

- **Build_Pipeline**: The GitHub Actions CI/CD workflow that compiles, packages, tests, and publishes container images
- **Dockerfile**: A multi-stage build specification that produces a deployable container image for a Java service
- **AOT_Compilation**: Spring Boot Ahead-of-Time compilation that pre-computes bean definitions and configuration at build time to reduce startup latency
- **Lambda_Web_Adapter**: The AWS Lambda Web Adapter (`aws-lambda-web-adapter`), a sidecar layer that translates Lambda invoke events into HTTP requests on a configurable port
- **RIE**: The AWS Lambda Runtime Interface Emulator, a local proxy that simulates the Lambda invoke API so container images can be tested locally without deploying to AWS
- **Pact_Broker**: A service or file-based store that holds consumer-driven contract (Pact) files shared between consumer and provider verification
- **Consumer_Contract**: A Pact file generated by the frontend (consumer) that declares the expected request/response shape for a given API endpoint
- **Provider_Verification**: The process where a backend service replays consumer contracts against its live API to confirm compatibility
- **Custom_JRE**: A minimal Java runtime produced by `jlink` containing only the modules required by Spring Boot (e.g., `java.base`, `java.sql`, `java.naming`, `java.instrument`, `java.management`), reducing image size and memory footprint
- **Deep_Health_Check**: A readiness probe that validates end-to-end connectivity by routing through the API_Gateway to a downstream service endpoint (e.g., `GET /api/portfolio/health`), confirming that both the gateway and its route targets are initialized
- **Health_Check**: A Docker or HTTP-based readiness probe (e.g., `/actuator/health`) that signals a container is ready to accept traffic
- **Verification_Suite**: The combined set of Pact contract tests and Playwright E2E tests that validate a container image before deployment
- **API_Gateway**: The Spring Cloud Gateway service (`api-gateway`) that routes all external HTTP traffic to backend services on port 8080
- **Portfolio_Service**: The Spring Boot service that manages portfolio holdings and valuations, exposed via `/api/portfolio/**`
- **Insight_Service**: The Spring Boot service that generates AI-powered investment insights, exposed via `/api/insights/**`
- **Container_Image**: An OCI-compliant Docker image produced by the Build_Pipeline and published to GHCR
- **GHCR**: GitHub Container Registry, the OCI artifact store used by the project

## Requirements

### Requirement 1: Multi-Stage Dockerfile for Java Services

**User Story:** As a platform engineer, I want a multi-stage Dockerfile for each Java service that uses Gradle to build, applies Spring Boot AOT compilation, and produces a minimal final image with the AWS Lambda Web Adapter and RIE, so that the same container image runs locally for testing and deploys to AWS Lambda without modification.

#### Acceptance Criteria

1. WHEN the Dockerfile is built, THE Dockerfile SHALL use a Gradle-compatible JDK 25 base image in the build stage to compile the service source code and produce a Spring Boot executable JAR
2. WHEN the build stage executes, THE Dockerfile SHALL invoke Spring Boot AOT processing (`-Dspring.aot.enabled=true` or the `processAot` Gradle task) so that bean definitions and configuration are pre-computed at build time
3. THE Dockerfile SHALL include a `jlink` stage that uses `jdeps` to dynamically analyze the Spring Boot JAR and its dependencies to determine the exact set of required Java modules, then produces a Custom_JRE containing only those modules (which may include `java.base`, `java.sql`, `java.naming`, `java.instrument`, `java.management`, `java.desktop`, `jdk.unsupported`, `java.security.jgss`, and others required by Kafka, Auth, or Spring internals), so that the final image excludes unused JDK modules and remains well under the Lambda 10 GB image size limit
4. THE Dockerfile SHALL use a minimal runtime base image (Alpine or equivalent) in the final stage that copies only the Custom_JRE and the compiled application, without a full JDK or JRE installation
5. THE Dockerfile SHALL copy the AWS Lambda Web Adapter binary into the final image and configure it to forward HTTP traffic to port 8080
6. THE Dockerfile SHALL include the AWS Lambda Runtime Interface Emulator (RIE) binary in the final image so the container can be invoked locally using the Lambda invoke API
7. THE Dockerfile SHALL set the ENTRYPOINT to execute the Lambda Web Adapter as the primary process
8. WHEN the container is started locally with `docker run -p 8080:8080`, THE Container_Image SHALL serve HTTP requests on port 8080 via the Lambda Web Adapter forwarding to the embedded Spring Boot server
9. WHEN the container is started with the RIE entrypoint override, THE Container_Image SHALL accept Lambda invoke requests on port 9000 and translate them to HTTP requests against the application

### Requirement 2: Gradle Build Integration for AOT and Docker

**User Story:** As a developer, I want the Gradle build to support AOT compilation and Docker image creation from the root project, so that I can build production-ready container images with a single command.

#### Acceptance Criteria

1. WHEN `./gradlew :portfolio-service:bootJar` is executed with AOT enabled, THE Build_Pipeline SHALL produce a Spring Boot JAR that includes AOT-generated metadata
2. WHEN `./gradlew :api-gateway:bootJar` is executed with AOT enabled, THE Build_Pipeline SHALL produce a Spring Boot JAR that includes AOT-generated metadata
3. WHEN `./gradlew :market-data-service:bootJar` is executed with AOT enabled, THE Build_Pipeline SHALL produce a Spring Boot JAR that includes AOT-generated metadata
4. WHEN `./gradlew :insight-service:bootJar` is executed with AOT enabled, THE Build_Pipeline SHALL produce a Spring Boot JAR that includes AOT-generated metadata
5. THE Build_Pipeline SHALL retain the existing `./gradlew build` behavior so that unit and integration tests continue to pass without modification

### Requirement 3: Pact Consumer Contract Tests for Frontend

**User Story:** As a frontend developer, I want consumer-driven contract tests for the `/api/portfolio` and `/api/insights/market-summary` endpoints, so that I can detect breaking API changes before they reach integration testing.

#### Acceptance Criteria

1. WHEN the frontend Pact tests execute, THE Verification_Suite SHALL generate a Consumer_Contract for the `/api/portfolio` endpoint that declares the expected request method, path, headers, and response body schema
2. WHEN the frontend Pact tests execute, THE Verification_Suite SHALL generate a Consumer_Contract for the `/api/insights/market-summary` endpoint that declares the expected request method, path, headers, and response body schema
3. THE Verification_Suite SHALL store generated Pact files in a deterministic, well-known path within the repository (e.g., `frontend/pacts/`) that is easily discoverable by both the CI/CD pipeline (via explicit path reference in the GitHub Actions workflow) and backend provider verification tests (via relative path or environment variable), so that files can be committed and shared without manual coordination
4. WHEN a Consumer_Contract field is removed or its type is changed in the provider response, THE Provider_Verification SHALL fail and report the specific incompatibility
5. THE Verification_Suite SHALL structure the Pact file storage and test configuration so that it is compatible with a future Pact_Broker integration, allowing migration from file-based sharing to a centralized broker without rewriting consumer or provider test code

### Requirement 4: Pact Provider Verification for Backend Services

**User Story:** As a backend developer, I want the portfolio-service and insight-service to verify consumer contracts during their test phase, so that I am alerted immediately when a code change breaks the API contract.

#### Acceptance Criteria

1. WHEN the portfolio-service test suite runs, THE Provider_Verification SHALL replay the `/api/portfolio` Consumer_Contract against the live portfolio-service API and confirm all interactions pass
2. WHEN the insight-service test suite runs, THE Provider_Verification SHALL replay the `/api/insights/market-summary` Consumer_Contract against the live insight-service API and confirm all interactions pass
3. IF a provider response does not match the Consumer_Contract, THEN THE Provider_Verification SHALL fail the test with a descriptive error identifying the mismatched field, expected value, and actual value
4. THE Provider_Verification SHALL use provider states to set up required test data (e.g., seeded portfolio holdings, cached market summary) before replaying each interaction

### Requirement 5: Playwright Container Health-Check Hardening

**User Story:** As a QA engineer, I want Playwright to wait for the Docker/RIE container to report healthy before executing E2E tests, so that tests do not fail due to container startup delays.

#### Acceptance Criteria

1. WHEN the Playwright test suite is launched against a containerized backend, THE Verification_Suite SHALL poll a Deep_Health_Check endpoint routed through the API_Gateway to a downstream service (e.g., `GET /api/portfolio/health` or equivalent) until it returns HTTP 200, confirming that both the gateway and its backend route targets are fully initialized before executing any test
2. IF the Deep_Health_Check endpoint is unavailable, THE Verification_Suite SHALL fall back to polling the API_Gateway shallow Health_Check endpoint (`/actuator/health`) until it returns HTTP 200
3. WHEN the Health_Check does not return HTTP 200 within a configurable timeout (default 120 seconds), THE Verification_Suite SHALL abort the test run and report a clear timeout error with the last observed Health_Check response
4. THE Verification_Suite SHALL use a Playwright `globalSetup` script or equivalent mechanism to perform the Health_Check polling, so that individual test files do not duplicate readiness logic
5. WHILE the Health_Check polling is in progress, THE Verification_Suite SHALL log each poll attempt with a timestamp and HTTP status code so that startup timing is observable in CI logs

### Requirement 6: Refactored GitHub Actions CI Pipeline

**User Story:** As a platform engineer, I want the GitHub Actions CI pipeline to build the multi-stage Docker image and run the full Playwright verification suite against that image, so that every PR is validated against the exact artifact that will be deployed.

#### Acceptance Criteria

1. WHEN a pull request is opened or updated, THE Build_Pipeline SHALL build the multi-stage Container_Image for each Java service using `docker build`
2. WHEN the Container_Images are built, THE Build_Pipeline SHALL start the services using `docker run` (or Docker Compose) with the same environment variables and port mappings used in production
3. WHEN the containerized services are running, THE Build_Pipeline SHALL execute the Playwright E2E test suite against the running containers
4. IF any Playwright test fails, THEN THE Build_Pipeline SHALL fail the workflow, upload the Playwright HTML report as a GitHub Actions artifact, and print container logs for debugging
5. WHEN all Playwright tests pass, THE Build_Pipeline SHALL tag the verified Container_Image and push it to GHCR
6. THE Build_Pipeline SHALL run Pact provider verification as part of the CI pipeline before the Playwright E2E stage so that contract violations are caught early
7. THE Build_Pipeline SHALL cache Gradle dependencies and Docker layers across workflow runs to minimize build time

### Requirement 7: Docker Compose Integration for Verification

**User Story:** As a developer, I want a Docker Compose configuration that starts all containerized services (using the new multi-stage images) alongside infrastructure dependencies, so that the verification suite can run against a realistic local environment.

#### Acceptance Criteria

1. THE Docker Compose configuration SHALL define services for all Java backends (portfolio-service, market-data-service, insight-service, api-gateway) using the new multi-stage Dockerfile
2. THE Docker Compose configuration SHALL include Health_Check definitions for each Java service so that dependent services and test runners can wait for readiness
3. THE Docker Compose configuration SHALL expose the API_Gateway on port 8080 and the frontend on port 3000, matching the existing local development port assignments
4. WHEN `docker compose up -d` is executed, THE Docker Compose configuration SHALL start all infrastructure dependencies (PostgreSQL, MongoDB, Kafka, Redis) and all Java services in the correct dependency order
5. THE Docker Compose configuration SHALL remain backward-compatible with the existing `docker-compose.yml` port mappings and environment variable names so that existing developer workflows are not disrupted
6. THE Docker Compose configuration SHALL define resource limits (`deploy.resources.limits.memory: 512M`) for each Java service to prevent a single Spring Boot instance from consuming all available host RAM during local verification runs

### Requirement 8: Local Developer Verification Workflow

**User Story:** As a developer, I want to run the full verification suite locally with a single command, so that I can validate changes before pushing to CI.

#### Acceptance Criteria

1. WHEN a developer runs the local verification command, THE Verification_Suite SHALL build all Container_Images, start the Docker Compose stack, wait for Health_Checks, execute Pact provider verification, and run the Playwright E2E suite in sequence
2. WHEN the verification completes, THE Verification_Suite SHALL print a summary indicating the pass/fail status of each verification stage (build, contracts, E2E)
3. IF any stage fails, THEN THE Verification_Suite SHALL stop execution, print the failure details, and leave container logs accessible for debugging
4. WHEN the verification completes (pass or fail), THE Verification_Suite SHALL provide a teardown command or automatic cleanup option to stop and remove all containers
